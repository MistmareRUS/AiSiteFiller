## 🎯 1. ГЛАВНАЯ БИЗНЕС-ЦЕЛИ И СТРАТЕГИЯ
* **Цель**: Создание полностью автономного пассивного дохода через построение сетки контентных сайтов под Рекламную сеть Яндекса (РСЯ) и CPA-партнерки.
* **Принцип**: Стратегия "Запостил и забыл". Мы используем только те площадки, которые дают бесплатный поисковый трафик из Яндекса и Google за счет своего траста (Domain Authority). Мы НЕ занимаемся ручным продвижением каналов, закупкой рекламы или ботоводством.
* **Экономика конвейера**: Наш ПК генерирует статьи через OpenAI API + уникальные картинки через локальный Stable Diffusion. Чтобы минимизировать расходы на токены (около 30 копеек за статью), все готовые материалы (HTML, текст, мета) навсегда кэшируются в MongoDB (`MongoDbStorageService`).

## 🛠️ 2. АРХИТЕКТУРА И СТЕК ПРИЛОЖЕНИЯ
Проект написан на **C# (Windows Forms)** и строго придерживается принципов **Чистой Архитектуры (Clean Architecture)**:
* `Domain` — Ядро бизнес-логики и сущностей.
* `Application / Interfaces` — Сценарии использования и контракты сервисов.
* `Infrastructure` — Техническая реализация (БД, OpenAI, интеграции).

### Веерная дистрибуция (Слой Infrastructure/Services):
1. **WordPressPublisherService** — Пуш статей на основной сайт под РСЯ. Лидер по пассивному SEO-трафику.
2. **DzenPublisherService** — Интеграция с Дзеном. Дает органические рекомендации и поисковый трафик Яндекса.
3. **TeletypePublisherService** — Блоговая платформа с быстрой индексацией. Работает как SEO-донор для основного сайта.
4. **Telegram / Vk / Ok PublisherServices** — ЭТИ СЕРВИСЫ УЖЕ НАПИСАНЫ, НО НАХОДЯТСЯ В РЕЖИМЕ "ПАУЗЫ" / "ЗАМОРОЗКИ". Мы используем их исключительно как доноров внешних ссылок (социальных сигналов) для прокачки основного WordPress-сайта. Никакого ботоводства, прокси и накруток лайков для них мы НЕ пишем.

## 📈 3. ТЕКУЩИЙ ВЕКТОР РАЗВИТИЯ И МАСШТАБИРОВАНИЕ
Для выхода на целевой уровень дохода система развивается по двум направлениям:
1. **Интеграция с VC.ru (`VcPublisherService.cs`)**: Добавление новой автономной платформы строго по аналогии с `TeletypePublisherService`. Нам нужен VC.ru для мгновенного залета статей в топ Яндекса и слива трафика на РСЯ-сайт или CPA-офферы.
2. **Мульти-сайтовость (Multi-site)**: Модернизация планировщика `ContentPlannerService` и `WordPressPublisherService` для веерной рассылки сгенерированного контента сразу на 5-10 разных сайтов/доменов пользователя на хостинге Beget.

## ⚠️ 4. ТЕХНИЧЕСКИЕ ОГРАНИЧЕНИЯ (КРИТИЧЕСКИ ДЛЯ ИИ)

* **Формат ссылок (Защита от багов парсинга чата и UI)**:
  - **В коде**: Любые URL собирать СТРОГО через атомарную конкатенацию строк. Цельных URL-строк в коде быть не должно! Пример: `string url = "https://" + "dzen." + "ru/" + "profile";`
  - **В ответах чата пользователю**: Разделять элементы ссылок символами "плюс с пробелами" (пример: `https:// + vc. + ru + /api`), иначе интерфейс ломает и обрезает ссылки.
- **Кириллица в Teletype**: Обязательное использование `Uri.EscapeDataString()` для слагов с кириллицей, чтобы визуальный редактор Teletype не ломал `<a>` теги.
- **Маскировка (Beget .htaccess)**: Запрос у пользователя структуры `.htaccess` с редиректом `^go/(.*)$` на `market.yandex.ru` при работе с CPA.
* **Внешняя инфраструктура (Beget)**:
  - На хостинге **beget.com** крутятся сайты, базы данных и легковесные PHP-скрипты для приема картинок и POST-запросов от приложения. Эти файлы НЕ хранятся в репозитории Git. 
  - *Правило для ИИ*: При любых изменениях в логике отправки, всегда запрашивай у пользователя актуальную структуру скриптов на Beget, чтобы не нарушить их интеграцию.




## 🌐 5. ЗАДЕЛ ПОД ПАРАЛЛЕЛЬНЫЙ БУРЖ (МЕЖДУНАРОДНОЕ МАСШТАБИРОВАНИЕ)
Вся текущая разработка RU-сегмента должна вестись с учетом архитектурного задела (Extensibility) под параллельный запуск валютных Бурж CPA-сеток. Чтобы потом не переписывать код с нуля, при реализации модулей строго соблюдать следующие правила:

*   **Локализация контента (Multi-Language)**:
    - Готовность `OpenAiGptService` к переключению языкового контекста (RU/EN) для моделей и промптов. 
    - Архитектура должна поддерживать передачу языкового контекста (RU/EN) для переключения языковых моделей и характера промптов (информационный под РСЯ / продающий обзор-сравнение под CPA).
*   **CPA-маркеры с `|||`**: Использование формата `[CPA_LINK_PLACEHOLDER:|||БРЕНД_И_МОДЕЛЬ|||]` для защиты от ложных срабатываний и кодирования данных через `CpaLinkHelper.cs`.
*   **Гибкость очередей в `ContentPlannerService`**:
    - Планировщик должен быть готов к разделению потоков публикаций: "Поток RU" (WordPress, Дзен, Teletype) и будущий "Поток US" (Medium, Substack и др.).
    - Задачи в очереди должны иметь флаг локализации или идентификатор целевой языковой группы, чтобы исключить путаницу контента.
*   **Абстракция Издателей (`IPublisherService`)**:
    - Интерфейсы публикации должны оставаться максимально чистыми и универсальными, чтобы добавление условного `MediumPublisherService` происходило простым копированием и адаптацией по аналогии с `TeletypePublisherService`, без изменения ядра системы.
*   **Кэширование в MongoDB**:
    - Структура документов в `MongoDbStorageService` должна содержать поле языка (`Language`) или региона, чтобы база данных корректно разделяла кэшированные статьи на русском и английском языках.

Если точный код файла (например, `MainForm.cs` или фонового воркера) не загружен в текущую сессию — ИИ обязан остановиться и запросить прямую ссылку или текст файла у пользователя.
